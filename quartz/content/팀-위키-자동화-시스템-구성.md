# 팀 위키 자동화 시스템 개요 및 작동 원리

팀 위키 자동화 시스템 구축이 완료되었습니다. 이 시스템은 팀의 지식 관리 효율성을 높이고, 문서화 프로세스를 자동화하여 생산성을 향상시키는 것을 목표로 합니다.

## 주요 구성 요소

팀 위키 자동화 시스템은 다음과 같은 핵심 구성 요소로 이루어져 있습니다.

*   **[[Gemini Flash API]]**: 수집된 원시 문서의 내용을 분석하고, 위키 형식에 맞게 정보를 구조화하여 간결하고 명확한 페이지를 생성합니다. 또한, 기존 위키 페이지 목록을 참고하여 내용 통합 또는 신규 페이지 생성을 판단하고, 관련된 위키링크를 자동으로 제안하여 지식 간의 연결성을 강화합니다.
*   **[[GitHub Actions]]**: 프라이빗 저장소의 문서 변경을 감지하고, Gemini API를 통해 문서를 처리하며, 생성된 위키 페이지를 퍼블릭 저장소로 동기화하고, 최종적으로 정적 사이트를 빌드하여 배포하는 일련의 자동화된 워크플로우를 실행합니다. 문서 정리, 웹사이트 배포 등 전체 워크플로우를 자동화하는 CI/CD 파이프라인 역할을 수행합니다.
*   **[[Quartz]]**: Markdown 파일을 정적 HTML 웹사이트로 변환하는 도구입니다. 이 시스템에서는 생성된 위키 페이지를 사용자 친화적인 웹 환경으로 게시하는 데 사용됩니다.
*   **[[Obsidian]]**: 팀원이 로컬에서 위키 문서를 작성하고 관리할 수 있도록 지원하는 지식 관리 도구입니다. `.obsidian/` 폴더를 저장소에 포함하여, Git 플러그인 설정까지 완료된 상태로 팀원들이 쉽게 공동 작업할 수 있도록 합니다. 팀원들이 개인 및 팀 노트를 자유롭게 작성하고 관리하며 위키 문서를 생성하는 주요 환경이자 도구입니다.
*   **[[GitHub Secrets]]**: Gemini API 키, Public 저장소 접근을 위한 Personal Access Token (PAT) 등 민감한 인증 정보를 안전하게 관리하고 자동화 워크플로우에서 활용할 수 있도록 합니다.

## 작동 원리 및 자동화 워크플로우

팀 위키 자동화 시스템은 원시 문서 작성부터 최종 웹사이트 게시까지 전체 과정을 자동화하여 효율적인 지식 관리를 지원합니다. 주요 워크플로우는 다음과 같이 구성됩니다.

### 1. 프라이빗 저장소에서의 문서 생성 및 처리

1.  **원시 문서 작성**: 팀원들은 `raw/` 폴더에 마크다운 형식으로 회의록, 메모 등 자유로운 형식의 문서를 작성합니다.
2.  **GitHub Push**: 작성된 문서는 GitHub 프라이빗 저장소에 푸시됩니다.
3.  **GitHub Actions 트리거**: 푸시 이벤트 발생 시 `.github/workflows/wiki-gen.yml` GitHub Action이 자동으로 실행됩니다. 이 액션은 우분투 컨테이너 환경을 설정하고 필요한 파이썬 라이브러리를 설치합니다.
4.  **문서 처리 스크립트 실행**: `scripts/generate_wiki.py` 스크립트가 실행됩니다. 이 스크립트는 `raw/` 폴더 내의 마크다운 파일을 스캔하여 새로 추가되거나 변경된 문서를 식별합니다 (파일 해시 비교).
5.  **[[Gemini Flash API]] 연동**: 식별된 문서는 Gemini Flash 2.5 API로 전송되어 위키 페이지 형식으로 정리됩니다. 이때, 기존 위키 페이지 목록을 참고하여 내용 통합 또는 신규 페이지 생성을 결정하도록 지시합니다.
6.  **위키 페이지 생성**: Gemini는 페이지 제목, 대상 파일명, 새 페이지 여부 (`is_new`), 그리고 정리된 본문을 포함하는 JSON 형식의 응답을 반환합니다. 스크립트는 이 응답을 파싱하여 `wiki/` 폴더에 해당 파일을 저장합니다.
7.  **내부 링크 생성 및 인덱스 업데이트**:
    *   관련 개념 간의 [[파일명]] 형식 위키링크를 자동으로 삽입합니다. 이는 [[Obsidian]]과 [[Quartz]] 모두에서 인식 가능한 문법입니다.
    *   `index.md` 파일도 [[Gemini Flash API]]를 통해 카테고리별로 묶어 재생성됩니다.
8.  **프라이빗 저장소 커밋 및 푸시**: 변경된 `wiki/` 폴더의 내용은 GitHub Action에 의해 자동으로 커밋되고 프라이빗 저장소에 푸시됩니다.

### 2. 퍼블릭 저장소로의 배포 및 정적 사이트 생성

1.  **퍼블릭 저장소 클론**: 이어서 GitHub Action은 퍼블릭 저장소를 임시 폴더에 클론합니다.
2.  **파일 동기화**: `rsync` 명령어를 사용하여 프라이빗 저장소의 `wiki/` 내용을 퍼블릭 저장소의 `quartz/content/` 디렉터리로 복사합니다. 이때, Private 저장소가 Public 저장소에 접근하기 위한 권한 부여를 위해 [[GitHub Secrets]]에 등록된 `PUBLIC_REPO_TOKEN` (Personal Access Token)이 사용됩니다.
3.  **정적 사이트 빌드 트리거**: 퍼블릭 저장소에 변경 사항이 푸시되면, 해당 저장소의 `deploy.yml` GitHub Action이 다시 실행됩니다.
4.  **[[Quartz]] 빌드**: 이 액션은 `npm ci` 및 `npx quartz build` 명령어를 통해 Quartz 정적 사이트 빌드를 수행합니다. 빌드된 HTML 파일들은 `quartz/public/` 폴더에 생성됩니다.
5.  **GitHub Pages 배포**: 생성된 정적 사이트 파일들은 GitHub Pages로 업로드되어 최종적으로 `hooni0918.github.io/MINO-WIKI-PUBLIC/` 주소를 통해 접근 가능하게 됩니다.

## 시스템 특징 및 효율성

*   **전체 자동화**: `raw/` 폴더에 문서를 작성하고 푸시하는 것만으로 전체 배포 과정이 2~3분 내에 자동으로 완료됩니다.
*   **보안 강화**: [[Gemini Flash API]] 키 및 [[GitHub Secrets]]에 등록된 Personal Access Token(PAT)과 같은 민감 정보는 철저히 관리되며, 코드에 하드코딩되지 않습니다.
*   **[[Obsidian]] 연동**: `.obsidian/` 폴더를 저장소에 포함하여, 팀원이 저장소를 클론한 후 Obsidian으로 열면 Git 플러그인 설정까지 완료된 상태로 즉시 위키를 관리할 수 있습니다.
